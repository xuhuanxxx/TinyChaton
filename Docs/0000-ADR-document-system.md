# 文档系统架构决策记录

## 问题/目标

项目文档混乱无序，缺少统一规范导致无法追踪问题状态、关联关系和时间线。需要建立可维护的文档系统。

## 背景

### 现有问题
1. 文档分类混乱 - 审计报告/TODO/测试计划/架构分析混杂
2. 命名不一致 - 时间戳格式、前缀随意
3. 无元数据 - 状态、优先级、创建/更新时间缺失
4. 无追踪机制 - 问题解决状态、负责人、文档关联不清楚
5. 无索引 - 找不到活跃文档和已归档内容

### 约束条件
- 扁平化目录结构，避免多层嵌套
- 文件名体现分类，不依赖目录层级
- 便于人工阅读和 AI 处理
- 支持文档间引用和追踪

## 决策内容

### 1. 命名规范

**格式**: `NNNN-[类型]-<主题>.md`

- **序号**: 4位数字（0001起），按创建时间递增，不回收
- **类型**: 固定枚举值
  - `TODO` - 待实施任务
  - `AUDIT` - 代码审计报告
  - `ARCH` - 架构设计
  - `SPEC` - 功能规格
  - `TEST` - 测试计划
  - `ISSUE` - 问题记录
  - `ADR` - 架构决策记录
- **主题**: 英文短语，连字符分隔，简洁明确

**理由**
- 序号前缀保证排序稳定，便于引用（"见 #0042"）
- 类型前缀快速识别文档性质
- 日期放元数据而非文件名，避免重命名

### 2. 元数据结构

```yaml
---
id: NNNN              # 与文件名序号一致
priority: P0|P1|P2|P3 # 优先级
created: YYYY-MM-DD   # 创建日期
updated: YYYY-MM-DD   # 最后更新日期
assignee: <optional>  # 负责人（可选）
relates: [#0001, #0003] # 关联文档序号
status: ACTIVE|RESOLVED|ARCHIVED|DRAFT # 状态放最后
---
```

**理由**
- YAML 格式易解析
- `status` 放最后便于快速定位
- `relates` 用序号引用建立文档网络
- `updated` 强制更新追踪变更历史

### 3. 内容结构

```markdown
# 标题

## 问题/目标
<1-2句话说明为什么写这个文档>

## 内容
...

## 结论/下一步
<必须明确: 解决了什么/还需要做什么/待验证事项>
```

**理由**
- 结论优先原则：开头说目的，结尾说行动
- 强制明确后续动作，避免文档沉睡
- 结构固定降低认知负担

### 4. 文档类型约束

| 类型 | 必须包含 | 典型场景 |
|------|---------|---------|
| TODO | 验收标准、实施步骤 | 待实施的开发任务 |
| AUDIT | 风险等级、代码位置 | 代码审计发现的问题 |
| ARCH | 不变量、权衡点 | 架构设计与决策 |
| SPEC | 行为边界、异常处理 | 功能规格定义 |
| TEST | 前置条件、通过标准 | 测试计划与回归检查 |
| ISSUE | 复现路径、影响范围 | 问题追踪与分析 |
| ADR | 决策背景、方案对比、选择理由 | 重大架构决策记录 |

**理由**
- 每种类型有明确验证要素
- 防止文档内容发散和缺失关键信息

### 5. 序号管理

- 维护 `Docs/.next-id` 文件存储下一个可用序号
- 创建新文档时读取并递增
- 已删除文档序号不回收，保持历史连续性

**理由**
- 单一真相源，避免序号冲突
- 简单文本文件，易于版本控制和手动修改

### 6. 写作准则

- **结论优先**: 开头直接说明目的，结尾明确后续动作
- **可验证性**: 所有问题必须可复现，所有方案必须可验证
- **无歧义**: 使用明确术语，引用具体代码位置或配置项
- **避免过时**: 不记录易变实现细节，记录稳定原则与约束
- **关联追踪**: 重大变更必须关联相关审计/TODO/测试文档

**理由**
- 与项目 AGENT.md 其他准则一致
- 确保文档长期价值和可维护性

## 拒绝的方案

### 方案 A: 多层级目录结构
```
Docs/
├── architecture/
├── tasks/
├── audits/
└── testing/
```

**拒绝理由**: 增加路径复杂度，分类已在文件名体现

### 方案 B: 日期作为文件名一部分
```
TODO-ACTIVE-2026-03-02-chat-taint.md
```

**拒绝理由**: 日期是元数据而非标识符，放文件名会导致内容不变时也需要重命名

### 方案 C: 状态作为文件名一部分
```
0001-TODO-ACTIVE-chat-taint.md
```

**拒绝理由**: 状态频繁变化，会导致大量文件重命名，Git 历史混乱

## 影响范围

### 新建文档流程
1. 读取 `Docs/.next-id` 获取序号
2. 按 `NNNN-[类型]-<主题>.md` 创建文件
3. 填写完整元数据头部
4. 递增并更新 `.next-id`

### 现有文档
- 保持原样，不强制重命名
- 新文档遵循新规范
- 逐步迁移重要活跃文档

### 协作影响
- AI 创建文档自动遵循规范
- 人工创建需参考 AGENT.md 第 7 节
- 序号冲突通过 `.next-id` 避免

## 验证方式

- 新建 5 个不同类型文档测试命名和元数据
- 验证文档间引用（`relates` 字段）可追踪
- 确认按文件名排序时序号顺序正确
- 检查 `.next-id` 更新机制无冲突
