# AGENT.md - TinyChaton 全局开发约束

本文档用于指导 AI 在本项目中的改动行为。目标是提供长期稳定的高层规则，避免被短期实现细节误导。

## 1. 目标与边界

- TinyChaton 是轻量聊天增强插件，优先保证聊天体验稳定、可预期、可回退。
- 不引入与聊天主链路无关的复杂能力，不做"全能型 UI 框架"。
- 优先复用现有能力和注册表，不随意新增平行机制。

## 2. 架构与行为约束

### 架构不变量
- **策略驱动**：功能启停由策略层统一决定，不由各模块分散判断环境
- **生命周期可逆**：所有动态功能必须可完整 enable/disable/teardown
- **数据驱动**：配置与枚举以注册表为真相源，禁止散落硬编码。同一语义字段只允许一个当前 schema
- **入口收口**：关键路径走统一网关分发，不允许旁路。同一能力只能有一条主执行路径
- **启动规范化**：历史数据只在启动阶段规范化，运行期只处理当前结构与枚举

### 行为语义约束
- **防御性处理**：处理外部数据时必须验证类型与合法性
- **用户意图优先**：手动操作必须可用，自动化行为受策略约束

## 3. 变更准则

- 兼容不是默认美德，不为"可能存在的旧调用方"保留双路径。
- 删除胜于注释：确认废弃代码直接删除，不以注释保留。
- 避免隐式全局副作用（全局变量覆盖、不可恢复 hook、难追踪状态写入）。
- 依赖顺序由清单保证，业务模块不为"可能未加载"做兜底。

## 4. 最小工程规范

- `local` 优先，避免意外全局变量。
- 命名清晰一致：常量大写下划线，公开 API 用 `addon:Method()` 风格。
- 接口收口：减少可变参数与旧签名重载，统一到明确上下文对象。
- 显式默认优先：非法输入统一回默认或 no-op，不做意图猜测式修复。
- 高频路径注意性能：避免不必要分配、重复闭包与重复查找。
- 提交前至少做 Lua 语法检查：`luac -p <file.lua>` 或批量检查。

## 5. 文档规范

### 命名格式
```
NNNN-[类型]-<主题>.md
```

- 序号：4位数字（0001 起），手动分配，便于引用（如"见 #0042"）
- 类型：`TODO`/`AUDIT`/`ARCH`/`SPEC`/`TEST`/`ISSUE`/`ADR`
- 主题：英文短语，连字符分隔

**示例**
```
0001-TODO-chat-taint-pipeline.md
0002-AUDIT-event-filter-leak.md
0003-ARCH-settings-restructure.md
```

### 元数据头部（必需）
```markdown
---
id: NNNN
priority: P0|P1|P2|P3
created: YYYY-MM-DD
updated: YYYY-MM-DD
assignee: <optional>
relates: [#0001, #0003]
status: ACTIVE|RESOLVED|ARCHIVED|DRAFT
---
```

- `status` 必须放最后一行
- `relates` 用序号引用关联文档
- `updated` 每次修改必须更新

### 内容结构（必需）
```markdown
# 标题

## 问题/目标
<1-2句话说明为什么写这个文档>

## 内容
...

## 结论/下一步
<必须明确: 解决了什么/还需要做什么/待验证事项>
```

### 文档类型定义

- **TODO**: 待实施任务，必须包含验收标准与实施步骤
- **AUDIT**: 代码审计报告，必须标注风险等级与代码位置
- **ARCH**: 架构设计，必须说明不变量与权衡点
- **SPEC**: 功能规格，必须定义行为边界与异常处理
- **TEST**: 测试计划，必须包含前置条件与通过标准
- **ISSUE**: 问题记录，必须包含复现路径与影响范围
- **ADR**: 架构决策记录，必须说明决策背景、方案对比、选择理由

### 写作准则

- 结论优先：开头直接说明目的，结尾明确后续动作
- 可验证性：所有问题必须可复现，所有方案必须可验证
- 无歧义：使用明确术语，引用具体代码位置或配置项
- 避免过时：不记录易变实现细节，记录稳定原则与约束
- 关联追踪：重大变更必须关联相关审计/TODO/测试文档
